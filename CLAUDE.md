# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

PidgeIoT is an edge-native IoT platform built in Rust, structured as a Cargo workspace with three crates:

- **`dovecote`** (backend): serverless edge router on Cloudflare Workers + Durable Objects. Compiles to a `cdylib` via `worker-build`. Handles device ingestion, provisioning, and session validation.
- **`fancier`** (frontend): WebAssembly SPA built with Dioxus 0.7 + TailwindCSS/DaisyUI. The human-facing dashboard.
- **`capsules`** (shared models): serde structs/RPC schemas shared by both `dovecote` and `fancier` so frontend/backend stay in sync. Keep this crate free of Worker- or Dioxus-specific dependencies.

Auth/identity is handled by a self-hosted Ory Kratos instance (via `docker-compose.yml`) for dashboard users, plus per-pigeon Ed25519 keypairs and compact binary bearer tokens (not JWTs) for device auth — see `dovecote` below.

Full HTTP API reference (every dashboard + device route, auth models, request/response shapes): `docs/api.md`.

## Development commands

Three services run in parallel, each in its own terminal, from repo root unless noted:

```sh
# 1. Auth + DB (Kratos, Postgres, MailSlurper)
docker-compose -f docker-compose.yml up --force-recreate

# 2. Edge backend (dovecote) — served at http://127.0.0.1:8787
cd dovecote && bunx wrangler dev --ip 127.0.0.1 --port 8787 --env dev

# 3. Frontend (fancier) — served at http://127.0.0.1:4455
cd fancier && dx serve --addr 127.0.0.1 --port 4455

# Live CSS rebuild while developing fancier
cd fancier && bunx @tailwindcss/cli -i ./assets/tailwind.css -o ./assets/styling/main.css --watch
```

- Kratos Admin UI: http://127.0.0.1:3000
- MailSlurper (local email capture): http://127.0.0.1:4436

Rebuilding the architecture diagram (fancier only):
```sh
cd fancier && bunx mmdc -i assets/architecture.mmd -o assets/images/architecture.svg -b transparent
```

Standard Cargo workflows apply per-crate (`cargo check -p dovecote`, `cargo check -p fancier`, `cargo check -p capsules`) — `dovecote` and `fancier` both target wasm/Workers, so a plain `cargo build` at the workspace root will not fully validate them; use `wrangler dev`/`dx serve` (above) or `worker-build`/`dx build` for real compilation checks.

Formatting: `tab_spaces = 2` (see `rustfmt.toml` in root, `dovecote/`, and `fancier/`) — this repo uses 2-space indentation everywhere, not the Rust default of 4.

## Architecture

### `dovecote` — edge router + Durable Objects

- `src/lib.rs` — the single Cloudflare Worker entrypoint (`#[event(fetch, ...)]`). Defines all HTTP routes on a `worker::Router`. Every route handler manually attaches CORS via `.with_cors(&cors)` and returns explicit error `Response`s (no `?`-propagation through route closures — see git history: "Unroll router let chains to avoid silent failures", "Response::Error() can't fail, unwrap instead of ?"). When adding routes, follow the existing pattern of `let Ok(x) = ... else { return Response::error(...) }` rather than `?`.
- CORS is **not** a global static — each route closure calls `build_cors(&ctx.env, &req)` itself (`lib.rs`), matching the request's `Origin` header against `ROOT_URL` and handing `Cors::with_origins` exactly that one matching value (or `ROOT_URL` as an inert non-matching default). This exists because `worker::Cors` joins every configured origin into `Access-Control-Allow-Origin` with commas rather than matching per-request — a comma-joined value is invalid per the CORS spec and real browsers silently reject it once more than one origin is configured, so a single shared multi-origin `Cors` can't be used here. It also can't be computed once in `main` and shared by reference across routes, since each route is a separate `async move` closure and `Cors` isn't `Copy`. `ROOT_URL` (`[vars]`/`[env.dev.vars]`, `wrangler.toml`) is the frontend's own origin in both environments — `https://pidgeiot.com` in production, the local `dx serve` address in dev.
- Two local macros in `lib.rs` remove boilerplate: `get_pigeon_do!` (resolves `pigeon_id` → Durable Object stub) and `get_db!` (opens a Postgres client via Hyperdrive, mapping failure to a 500). Both take the caller's local `cors` as a final `$cors:expr` argument (needed for their early-return error paths) since macro hygiene means they can't see a caller-scope `cors` implicitly.
- **Dual persistence model**: each `Pigeon` is authoritative in its own Durable Object (SQLite-backed, one row per DO), and is also mirrored into Postgres (via Cloudflare Hyperdrive) for cross-pigeon querying (e.g. listing a flock's pigeons). The DO is the source of truth; every mutating route proxies to the DO first, then best-effort syncs the result to Postgres (`update_pigeon_pg_db`, `insert_pigeon_pg_db`, etc. in `src/helpers/pigeons.rs`) — a Postgres sync failure is logged but does not fail the request.
- `src/objects/pigeons.rs` — the `Pigeons` Durable Object. Owns its own embedded SQLite schema (created in `DurableObject::new`, mirrors `init-db.sql`'s Postgres schema but with SQLite triggers for `updated_at`/immutability/shadow version bumping). Internal routes are dispatched by string path match (`/pigeon/get`, `/pigeon/create`, `/pigeon/shadow/update`, etc.) and are reached only via `proxy_to_pigeon_do` from the top-level router — never exposed directly to the internet.
- Authorization inside the DO is header-based, not cookie-based, for user-facing routes: the router resolves the Kratos session into a user id and forwards it as `X-User-Id`; `is_authorized`/`is_owner` in `objects/pigeons.rs` check that header against the DO's local `pigeon_acl` table. The device-facing route (`GET /device/pigeons/:id/shadow`, dispatched inside the DO as `/pigeon/device/shadow`) does **not** go through `X-User-Id`/ACL at all — a device has no Kratos user identity. Instead `get_shadow_device` (`objects/pigeons.rs`) verifies the request's `Authorization` bearer token directly against this pigeon's own `device_public_key` column, via `verify_device_token` (`objects/helpers.rs`). This used to be gated by a `X-User-Id: ""` sentinel that `is_authorized` had no special case for, permanently 403ing the route — replacing ACL-based auth with real per-pigeon cryptographic auth for this route removed the need for that sentinel entirely.
- Device auth mechanism (`src/objects/helpers.rs`, `src/objects/pigeons.rs`) — **not JWT**: each pigeon gets its own Ed25519 keypair, generated fresh in the DO on `create` and on `token/refresh` (`mint_device_credential`). Only the public key is ever persisted (in the DO's own SQLite `pigeons.device_public_key` column — never mirrored to Postgres, never returned over the API); the private key signs one token and is discarded. The token itself is a 69-byte binary blob — `version(1) | expires_at(4, u32 LE unix seconds) | signature(64)` — base64url-encoded for transport, with no `pigeon_id`/subject claim: the binding to a specific pigeon comes from which pigeon's stored public key the DO verifies against, not from a claim inside the token. There are no `DEVICE_SEED`/`DEVICE_PUBLIC_KEY` Worker secrets anymore — verification happens entirely inside the owning DO (`verify_device_token`), reached via `proxy_to_pigeon_do` forwarding the caller's `Authorization` header straight through (`src/helpers/pigeons.rs`). **Refreshing a device's token revokes the previous one**: `refresh_token` mints a brand-new keypair and overwrites `device_public_key`, so the old token's signature can never verify again, regardless of its embedded expiry.
- `src/helpers/auth.rs` now holds only `authenticate_browser` (Kratos session cookie validation for dashboard users).
- **Device-facing shadow report-back and telemetry ingestion** (`objects/pigeons.rs`, added 2026-07-15): both share device auth via `is_authorized_device()` (factored out of what used to be three near-identical checks duplicated across `get_shadow_device`/`report_shadow_device`/`report_telemetry_device` — bearer token vs. this pigeon's own `device_public_key`, no ACL). `POST /device/pigeons/:id/shadow` → DO-internal `/pigeon/device/shadow/report` → `report_shadow_device`: a device confirms it applied `target_config` by writing its own `current_config`/`current_version` into `pigeon_shadow`; like the dashboard's `PUT /pigeons/:id/shadow`, the gateway route also best-effort syncs the result to Postgres via the existing `update_shadow_pg_db`, so Postgres's mirrored shadow state doesn't go stale for pigeons that report back on their own. `POST /device/pigeons/:id/telemetry` → DO-internal `/pigeon/device/telemetry` → `report_telemetry_device`: body is a flat JSON object of string key/value pairs, upserted one row per key into the DO's `pigeon_telemetry` table (`key` `PRIMARY KEY`, `value`, `reported_at`) — latest-value-per-key, not a time-series log; `GET /pigeons/:id/telemetry/latest` (dashboard) reads it straight through the DO.
- **Telemetry history + user-definable forwarding endpoint** (added 2026-07-17, `queue.rs` + `helpers/telemetry.rs`): telemetry writes ride a Cloudflare Queue (`pidgeiot-telemetry`, per-env producer/consumer split in `wrangler.toml` — a plain `wrangler deploy`, not `versions upload`, is required to actually register a consumer). The consumer reads the DO's `TelemetryWriteResult` and picks one of two paths per-pigeon: if the pigeon has no `telemetry_endpoint` configured, best-effort write into Postgres `pigeon_telemetry_history` (queryable via `GET /pigeons/:id/telemetry/history` and the flock-wide variant, `query_telemetry_history_for_pigeon`/`_for_flock` in `helpers/telemetry.rs`); if one *is* configured (`PUT`-able via `update_telemetry_endpoint_pg_db`/the `telemetry_endpoint` column, `PigeonTelemetryEndpointUpdateRequest` in `capsules`), forward instead as InfluxDB line protocol v2 (GreptimeDB-compatible) via `forward_line_protocol` — the two paths are mutually exclusive per report, not both. `auth_token` on a configured endpoint is stripped on reads, same convention as connector tokens. Non-numeric telemetry values are stored/history'd but intentionally excluded from line-protocol forwarding (they'd emit invalid lines) and don't appear in `fancier`'s graphable-key picker.
- **Device dictionary-log ingestion** (added 2026-07-17, `objects/pigeons.rs`): `POST /device/pigeons/:id/logs` (device-auth'd, `is_authorized_device()`, capped at `capsules::MAX_LOG_CHUNK_BYTES` = 16KiB per chunk) appends into a 200-chunk DO-local ring buffer (`PigeonLogChunk`/`PigeonLogChunkRow` in `capsules`, base64-stored since chunks are opaque binary — Zephyr `CONFIG_LOG_DICTIONARY_SUPPORT` dictionary-encoded logs, not plaintext); `GET /pigeons/:id/logs` (dashboard-auth'd) returns the buffer as base64 for client-side decode. No Postgres mirror (DO-local ring buffer only, by design — this is a rolling debug buffer, not a durable log store). Binary (non-UTF-8) device request bodies are proxied to the DO via `proxy_binary_to_pigeon_do` (`helpers/pigeons.rs`), distinct from the JSON-only `proxy_to_pigeon_do`.
- **Firmware storage + FOTA** (added 2026-07-18, task #23): per-flock firmware catalog in Postgres (`init-db.sql`) plus content-addressed image bytes in R2 (`pidgeiot-firmware`/`-staging` buckets, per-env bindings in `wrangler.toml`). `capsules::FirmwareTarget`/`FirmwareImage`/`FirmwareUploadQuery` (+ `MAX_FIRMWARE_BYTES`) define the catalog shape; a pigeon's shadow `target_config` can embed a `firmware` key (`FirmwareTarget`: version string + expected size + sha256) that a device diffs against its own running version to decide whether to start a download — the shadow PUT from the dashboard replaces `target_config` wholesale, so `fancier`'s `merge_firmware_target` helper exists specifically to patch just that key in without clobbering `log`/`telemetry_interval`/`reboot`. Device download is `GET /device/pigeons/:id/firmware/:version` (device-auth'd), HTTP Range-aware (chunked download, matches `pigeon`'s FOTA client) — dovecote streams straight from R2, sha256 isn't re-verified server-side per chunk (the device verifies the full image after assembly). Known gotcha (found via real hardware e2e, 2026-07-20): an R2-stored FOTA test/staging image is a separately-built-and-signed artifact from whatever's currently flashed as the "primary" image — if a device's token gets rotated *after* a test firmware image was built, that image still carries the stale token baked in via `CONFIG_PIGEON_TOKEN`, and will 401 on every request once booted into, even though the FOTA mechanism itself worked correctly. Any OTA test image needs rebuilding (not just re-uploading) after a token rotation.
- **WebSocket device endpoint** (added 2026-07-19, task #32, `objects/ws.rs` + `objects/pigeons.rs`): `GET /device/pigeons/:id/ws` (gateway route checks `Upgrade: websocket`, proxies via `proxy_websocket_to_pigeon_do` in `helpers/pigeons.rs`) accepts a persistent channel for non-cellular devices as an alternative to polling HTTPS. Uses the Durable Object **Hibernation API** (`State::accept_websocket_with_tags`, `get_websockets_with_tag`, `WebSocket::serialize_attachment`/`deserialize_attachment`, `DurableObject::websocket_message`/`_close`/`_error` — confirmed present in the `worker` crate v0.8.5) so an idle connection survives DO eviction without keeping the DO warm. Gated by `is_authorized_device()` before accept, same as the device HTTPS routes; one socket per pigeon enforced via the tagged `get_websockets_with_tag` lookup. Frame protocol (`objects/ws.rs`) handles `telemetry`/`shadow_report`/`ping` frames from the device, and the DO broadcasts a `shadow_update` frame over any open socket whenever the dashboard writes a new shadow — so a WS-connected device can react to config pushes without polling. Full frame shapes documented in `docs/api.md`. **Device-side client landed 2026-07-20 (task #33)** — `pigeon`'s `CONFIG_PIGEON_WS` client + `pigeon-examples` `wifi_init` sample, hardware-verified end-to-end on a real ESP32-C6 against `dovecote-staging` (WS connect, `shadow_update` push applied ~1s after a dashboard PUT, telemetry-over-WS, HTTPS fallback, and socket-steal reconnect/one-socket-per-pigeon recovery all confirmed on-device). See `~/pigeon`/`~/pigeon-examples` CLAUDE.md for the device side. Backend WS endpoint **promoted to prod 2026-07-22** — the same staging-verified `dovecote` binary is now deployed to production (`api.pidgeiot.com`); the WS gateway route responds and gates there. The remaining belt-and-suspenders check is a real device connecting over WS to `api.pidgeiot.com` (the device-side client was hardware-verified end-to-end against `dovecote-staging`, and prod runs the identical code).
- Connectors: a `Pigeon` has one `Connector` (`Https` or `Coap`), each carrying its own endpoint + token/PSK. Tokens are always stripped before returning a `Pigeon` from `GET` routes (`objects/pigeons.rs::get`/`get_detail`) — they're only ever returned on `create` or `token/refresh`. The CoAP connector speaks CoAP-over-TLS/TCP (RFC 8323, `coaps+tcp://`), not CoAP-over-DTLS/UDP — `COAP_ENDPOINT`/`build_coap_endpoint` (`objects/pigeons.rs`) emit `coaps+tcp://`, and `capsules::CoapConfig`'s PSK fields are named `tls_psk_identity`/`tls_psk_secret` accordingly. This matches the sibling `~/pigeon` Zephyr device library, which has no on-device UDP support and was already ahead on this — see its `CLAUDE.md`/`pigeon.h` for the device-side rationale.
- Pigeon deletion (`delete` in `objects/pigeons.rs`, dispatched at `/pigeon/delete`, gated by `is_owner`) wipes the DO's own tables — `pigeon_acl` explicitly (it has no FK to `pigeons`, since each DO holds exactly one pigeon's ACL) and `pigeons` itself (`pigeon_shadow` cascades via `ON DELETE CASCADE`). Durable Objects have no "delete yourself" API; emptying its storage is the idiomatic equivalent. The gateway's `DELETE /pigeons/:pigeon_id` route already proxied here and synced the deletion to Postgres before this handler existed — it just 404'd with nothing to dispatch to.
- **Gotcha**: `SqlCursor::one()` (`worker` crate) throws an uncaught JS exception — crashing the DO, not a catchable `Result::Err` — when a query returns zero rows. Every handler in `objects/pigeons.rs` used to assume the `pigeons`/`pigeon_shadow` tables always have exactly one row, which was safe only because nothing could ever leave them empty before `delete()` existed. Use `to_array()` and take the first element instead (see `one_row` in `objects/pigeons.rs`) for any query that could plausibly run against an already-deleted pigeon's DO — plain `.one()` is only safe immediately after an `INSERT ... RETURNING` in the same request.
- `wrangler.toml` defines two environments: default (production, `api.pidgeiot.com`) and `[env.dev]` (local, `127.0.0.1`, using `localConnectionString` for Hyperdrive instead of the production binding id).

### `capsules` — shared models

Plain serde structs with no Worker/Dioxus dependencies, consumed by both other crates via `capsules.workspace = true`. Notable pattern: every entity has a `*Row` variant (deserializes DB-native types, e.g. unix-epoch floats/integers for timestamps) and a public API variant (RFC 3339 `OffsetDateTime`), connected by a `From<XRow> for X` impl. When adding a new persisted field, update both variants and the conversion.

### `fancier` — Dioxus SPA

- `src/lib.rs` defines the full route table (`Route` enum, `Routable` derive) and the top-level `App` component. Routes under `#[layout(Wrapper)] #[layout(AuthGuard)]` require an authenticated Kratos session (checked client-side via `session_cookie_valid()` against `SESSION_COOKIE_NAME`); everything else (login, registration, marketing pages) is public.
- Global reactive state is plain Dioxus `Signal`s provided via context, not a store library: `Session { state: Signal<AuthState> }` for auth, `LocalSession { flocks, pigeons }` (both `Signal<HashMap<...>>`) as a client-side cache of fetched entities.
- `src/api/*` are the HTTP client functions (one file per resource, e.g. `pigeons.rs`, `flocks.rs`), all going through `fetch_json` in `src/api/helpers.rs` — a thin `web_sys`/`wasm_bindgen_futures` wrapper (not `reqwest`), always `credentials: include` so the Kratos session cookie rides along. API functions return `Option<T>` (fetch/parse failure just collapses to `None`, logged via `tracing::error!`) and, on success, write straight into the relevant `LocalSession` signal — callers re-read from context rather than the function's return value for cached fields.
- Config constants (`src/config.rs`: `KRATOS_BROWSER_URL`, `API_HOST`, `SESSION_COOKIE_NAME`) are baked in at **compile time** via `option_env!`, sourced from `.env.dev`/`.env.release` by `build.rs` (picks the file based on Cargo's `PROFILE`). To change these per-environment, edit the `.env.*` file, not `config.rs`.
- Ory Kratos self-service flows (login/registration/recovery/verification/settings) are driven by `ory-kratos-client-wasm`, rendered generically through `ory_form_builder`/`ory_error` components rather than bespoke forms per flow.
- `dx serve`/`dx build` are configured via `Dioxus.toml`; wrangler serves the built static assets in production (`fancier/wrangler.toml` points `[assets].directory` at the Dioxus release output).
- Most modals are native `<dialog>` elements toggled imperatively via `document::eval(...showModal()/close()...)` (e.g. `UpdatePigeonModal`, `EditShadowModal`). Modals that hold their own reset-sensitive state instead render conditionally from a signal in the parent (`if show_modal() { Modal { ... } }`), so opening always remounts the component fresh — `TokenReveal` (`views/pigeons.rs`, gated by `Signal<Option<(pigeon_id, token)>>`) and `DeletePigeonModal` (`views/pigeon.rs`, gated by `Signal<bool>`, the typed-name-to-confirm delete flow) both use this pattern rather than the native `<dialog>` one, since a stale `<dialog>` left open with prior input wouldn't reset on its own. `TokenReveal`'s `on_close` navigates to the new pigeon's detail route (added 2026-07-18, task #30) — creation now ends on the pigeon detail view instead of the list.
- **Telemetry graphs** (`views/pigeon.rs` area, added 2026-07-17): user-defined graph configs (which telemetry key(s), time range) persist in `localStorage`, not the backend — a graph is a client-side view over `GET /pigeons/:id/telemetry/history`. Renders Live/Empty/Preview states depending on whether history exists for the selected range; non-numeric telemetry keys (see dovecote's telemetry-forwarding note above) are excluded from the key picker entirely.
- **Device log viewer** (`components/log_viewer.rs`, added 2026-07-18, task #25): reads the base64 ring-buffer chunks from `GET /pigeons/:id/logs` and decodes/downloads them client-side as a `Blob` (`helpers/download.rs`, `web_sys` `atob` → object-URL anchor-click pattern) rather than server-side — the chunks are opaque Zephyr dictionary-encoded log data, decoded host-side by the `pigeon` repo's `log_parser.py`, not by `fancier` itself; this viewer's job is just getting the raw bytes to the user's machine intact.
- **Firmware upload/assign UI** (`components/firmware_modal.rs`, `api/firmware.rs`, `helpers/crypto.rs`, added 2026-07-18, task #25): uploads a signed firmware image to the flock's catalog (`api/helpers.rs` gained `fetch_bytes` alongside the existing JSON `fetch_json`, both routed through a shared `dispatch()`), computing the sha256 client-side via `SubtleCrypto` (`helpers/crypto.rs::sha256_hex`) before upload so it can be cross-checked against what the server stores. Assigning a firmware target to a pigeon uses `merge_firmware_target` (unit-tested) to patch just the shadow's `firmware` key rather than overwriting `target_config` wholesale — see dovecote's firmware note above for why that matters.
- **Shadow JSON file upload** (added 2026-07-19, task #26): `EditShadowModal` accepts a `.json` file as an alternative to hand-editing `target_config` in the live editor — same validation/submit path either way, just a different input source.
- **Connection-state indicator** (`helpers/connection_state.rs`, added 2026-07-19, task #31): a pure `classify()` function (Online/Stale/Offline/Unknown), self-calibrated from each pigeon's own `telemetry_interval` (2×/5× multipliers, with 5min/30min fallbacks and clock-skew tolerance when `telemetry_interval` is unset) — no new backend state, derived entirely from existing `pigeon_shadow.updated_at`. Caveat documented in-module: `updated_at` bumps on *any* shadow write (including a dashboard-initiated config push), not just a device report-back, so "Online" can be slightly optimistic right after an admin edits a shadow. Rendered via `ConnectionBadge`; uses `badge-ghost`/`.status` keyed to `--color-base-content` for the "Unknown" state specifically, not DaisyUI's `badge-neutral`/`status-neutral` — `--color-neutral` in this theme is literal white-on-white (light) / black-on-black (dark), an invisibility trap caught by screenshotting real compiled CSS in both themes.
- **Landing page + footer pages** (reworked 2026-07-17 through 2026-07-19): the "infrastructure" section (`partials/infrastructure.rs`) uses an inlined local SVG (`assets/images/infrastructure-edge.svg` via `include_str!`+`dangerous_inner_html`, not `asset!()`+`<img>`) so it can theme against the app's own `data-theme` attribute rather than only OS `prefers-color-scheme` — the same pattern `architecture.svg` already used. Footer "Product" links (`views/features.rs`, `documentation.rs`, `pricing.rs`, `api_reference.rs`) are real pages now, not dead links; `api_reference.rs` renders `docs/api.md` through `pulldown-cmark` rather than duplicating the API reference by hand, so that file staying accurate keeps both the docs and the in-app page accurate together.
- **WASM code-split investigated and rejected as unsafe** (task #10, 2026-07-22): Dioxus 0.7.9 / `dioxus-cli` 0.7.9 do have a real, built-in route-level code-split mechanism — enable the `wasm-split` feature on the `dioxus` dependency, pass `dx build`/`dx serve --wasm-split` (an "Experimental" CLI flag), and `dioxus-router`'s `Routable` derive automatically wraps *every* leaf route in a lazy loader + suspense boundary (`dioxus-router-macro`'s `RouteType::Leaf` codegen, gated by `dioxus_config_macros::maybe_wasm_split!`) — no manual `#[component(lazy)]` annotation needed, just a `SuspenseBoundary` ancestor (e.g. wrapping `Wrapper`'s `Outlet`). It builds cleanly and DOES shrink the initial bundle dramatically (measured: main chunk 3.56MB → 1.3MB raw / ~1.18MB → ~390KB gzip for the shared bundle, plus a small per-route chunk, e.g. ~78KB raw for a landing-page-sized route) and chunk files fetch correctly over the network (verified via CDP-driven headless Chromium, both direct-load and route-to-route). **However, it is not safe to ship**: for real (non-toy) page content it silently fails to render for an unpredictable subset of components — the split `.wasm` chunk fetches with a real HTTP 200, but the route's content never mounts (stays a bare `<!--placeholder-->`), with zero console errors, zero thrown exceptions, and no failed network requests to explain it. Bisected extensively against the actual `Index` route: a trivial route, `Home` alone (icons + `Link` + `Route` reference), and a handful of minimal single-file repro apps (2, then 21, synthetic routes, matching the exact `#[layout(Wrapper)] #[layout(AuthGuard)] ... #[end_layout]` structure) all rendered correctly — but `Features` (`partials/features.rs`, a deeper-nested icon grid, no `Link`/`Route` usage) reliably failed 100% of the time across multiple clean rebuilds, while `Home` reliably succeeded across multiple clean rebuilds. No root cause was found (suspected: a correctness bug in `wasm-split-cli`'s relocation-based call-graph reachability analysis, consistent with it being labeled "Experimental" — its own error strings reference parsing wasm relocations/`--emit-relocs`/DWARF debug info, a fragile foundation for arbitrary real-world Rust codegen). All changes from this investigation (the `wasm-split` feature flag, a `SuspenseBoundary` in `Wrapper`, `--wasm-split` in `scripts/build-release.sh`) were reverted — do not re-enable without a newer `dioxus-cli` release and a from-scratch re-verification of real page content (not just a toy route), not only a successful build.

## Conventions worth knowing

- 2-space indentation (`rustfmt.toml`), consistent across all crates.
- No `?`-based early return inside the Worker route closures in `dovecote/src/lib.rs` — use explicit `let Ok(x) = ... else { return ... }` so failures always produce a CORS-wrapped error `Response` rather than silently 500ing through the framework's generic error path.
- Every response returned from the top-level router must call `.with_cors(&cors)`, where `cors` is that route's own `build_cors(&ctx.env, &req)` result (see `dovecote` architecture notes above) — there is no shared/global CORS instance to reference.
- DB sync from a Durable Object back to Postgres is always best-effort/fire-and-log (`console_error!`), never blocking or failing the primary request.
